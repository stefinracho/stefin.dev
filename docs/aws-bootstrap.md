# AWS Bootstrap

This document outlines the method to bootstrap your AWS account for Infrastructure as Code (Terraform via GitHub Actions). 

## Prerequisites

* A brand new AWS Account.
* [AWS CLI](https://docs.aws.amazon.com/cli/) installed (version 2.32.0 or higher).
* [Terraform](https://developer.hashicorp.com/terraform) installed (version 1.10.0 or higher).
* A GitHub repository for your code.

## 1. Configure the GitHub Environment

Terraform will create an OIDC provider that restricts deployment access to a specific GitHub Environment. You must create this environment before configuring your Terraform variables.

1. Navigate to your GitHub repository in your web browser.
2. Go to **Settings** > **Environments**.
3. Click **New environment**.
4. Name the environment (e.g., `production`) and click **Configure environment**.

## 2. Prepare the Temporary IAM User

Before touching the CLI, you need an IAM user authorized to use the temporary browser-based login method.

1. Log into the AWS Management Console with your root account.
2. **Specify user details:** Navigate to **IAM** > **Users** and click **Create user**. Name it (e.g., `terraform-bootstrap`). Do not check "Provide user access to the AWS Management Console."
3. **Set permissions:** Click **Attach policies directly**:
    * `AdministratorAccess` (to create the infrastructure)
    * `SignInLocalDevelopmentAccess` (required to use the `aws login` command)
4. **Review and create:** Review and create the user.
5. Click your newly created user in the **User name** column.
6. Go to the **Security credentials** tab. 
7. Click **Enable console access**.
8. Select **Autogenerated password** then click **Enable console access**.
9. Save the **Console Password** because you will need it to login to the console.

## 3. Establish a Secure, Temporary Session

Ensure your AWS CLI is up to date (`aws --version`).

1. Open your local terminal and run:

   ```bash
   aws login
   ```

2. Your default web browser will open. Log in using the `terraform-bootstrap` credentials you just created. Do **NOT** login with root user credentials.
3. Once authenticated, your terminal will confirm the session is active. You now have temporary administrative credentials valid for 12 hours.

## 4. Configure the Bootstrap Variables

The Terraform code to bootstrap the account already exists in the `infra/bootstrap/` directory. 

1. Navigate to the bootstrap directory:

   ```bash
   cd infra/bootstrap
   ```

2. Copy the variables template:

   ```bash
   cp terraform.tfvars.example terraform.tfvars
   ```

3. Open `terraform.tfvars` and update the values for your new environment. (Ensure you provide a valid email address for `budget_alert_email` to receive billing alerts).

> [!IMPORTANT] 
> The `github_environment` variable should match the environment created in Step 1.

4. Copy the backend configuration template:

   ```bash
   cp config.s3.tfbackend.example config.s3.tfbackend
   ```

5. Open `config.s3.tfbackend` and update the values.

> [!IMPORTANT]
> The `bucket` and `region` values here must exactly match the `state_bucket_name` and `aws_region` values you set in `terraform.tfvars`.

## 5. The Local Apply

Because the S3 state bucket doesn't exist yet, we must temporarily build the infrastructure using local state.

1. Open `infra/bootstrap/main.tf` and temporarily **comment out** the `terraform` backend block at the top.
2. Initialize and apply:

   ```bash
   terraform init
   terraform apply
   ```

3. Review the plan and type `yes`. Terraform will create the S3 bucket, OIDC provider, and IAM roles using a local `terraform.tfstate` file.

## 6. The State Migration

Now that the secure S3 bucket exists in AWS, instruct Terraform to move its brain into it.

1. Open `infra/bootstrap/main.tf` and **uncomment** the `terraform` backend block.
2. Run the initialization command again, this time providing the partial configuration file:

   ```bash
   terraform init -backend-config="config.s3.tfbackend"
   ```

3. Terraform will detect the new S3 backend and the existing local state file. It will prompt you: 
   > *"Do you want to copy existing state to the new backend?"* 
4. Type `yes`. Terraform will securely transfer the state file directly into the S3 bucket.

## 7. Clean Up

The infrastructure is deployed, the state is secured remotely, and GitHub Actions is ready to take over via OIDC. Now, remove all manual access pathways.

1. **Destroy Local State:** Delete the local state files from your machine to ensure no plaintext secrets are left behind.

   ```bash
   rm terraform.tfstate terraform.tfstate.backup
   ```

2. **Save Role ARN:** Note the output of `github_actions_role_arn` from your console. You will need this for the next step.
3. **Terminate the Session:** Forcefully end your temporary 12-hour CLI session.

   ```bash
   aws logout
   ```

4. **Delete the User:** Log back into the AWS Console as root, navigate to IAM, and permanently delete the `terraform-bootstrap` user.

## 8. Finalize GitHub Actions Setup

To allow GitHub Actions to authenticate with AWS, it needs the ARN of the IAM role you just created. We will also configure deployment protections to ensure secure GitOps.

1. In your GitHub repository, navigate to **Settings** > **Environments**.
2. Select the environment you created in Step 1.
3. **Configure Deployment Protection Rules:**
    1. Check **Required reviewers** and add yourself (or your team). This ensures deployments pause for human approval.
    2. Uncheck **Allow administrators to bypass configured protection rules**.
    3. Click **Save protection rules**.
4. **Configure Branch Protections:**
    1. Under **Deployment branches and tags**, select **Selected branches and tags**.
    2. Add a rule to only allow the `main` branch.
5. **Add Environment Secret:**
    1. Under **Environment secrets**, click **Add environment secret**.
    2. Name the secret `AWS_ROLE_ARN`.
    3. Paste the `github_actions_role_arn` value from your Terraform apply output.
    4. Click **Add secret**.
6. **Add Repository Variables:**
    1. In the left sidebar of your repository settings, go to **Secrets and variables** > **Actions**.
    2. Click the **Variables** tab (next to the Secrets tab).
    3. Click **New repository variable**.
    4. Name: `AWS_REGION` | Value: (e.g., `us-west-2` or your chosen region).
    5. Click **New repository variable** again.
    6. Name: `STATE_BUCKET_NAME` | Value: (The name of the bucket you created in Step 5).

## 9. Commit & Push

Now that the remote state is secure and GitHub has the necessary authentication secret, you can safely commit your code. 

1. Ensure the `backend "s3"` block in your `main.tf` is uncommented.
2. Commit your `.tf` files and the newly generated `.terraform.lock.hcl` file, then push them to your GitHub repository. (Be sure *not* to commit `terraform.tfvars` or `config.s3.tfbackend`).

**Congratulations!** The local-to-remote bootstrap is complete. AWS is ready to securely receive infrastructure updates exclusively from GitHub Actions.
